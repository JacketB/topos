# 📋 План реализации: Количество техники (мультипликатор машин) в задачах планировщика

## 1. Проблема и математическая модель
В текущей реализации планировщика для каждой задачи рассчитывается работа одной машины выбранного типа (например, один экскаватор ЭОВ-4421). В реальности над отрывкой рва или котлованов может одновременно работать подразделение (отделение/взвод) инженерной техники, состоящее из нескольких машин.

### Математическое разделение:
1. **Суммарный ресурс (Машино-часы)**:
   Общая наработка техники в моточасах остается постоянной для выполнения заданного объема работ:
   $$\text{MachTotalHrs} = \frac{\text{Qty} \times \text{MachNorm} \times \text{PerfFactor}}{\text{Efficiency}}$$
   Этот параметр используется для ведомости расхода моторесурса и ГСМ.

2. **Календарная продолжительность этапа работы техники (Chronological Duration)**:
   При параллельной работе нескольких машин ($N_{\text{mach}}$) календарное время выполнения задачи сокращается пропорционально их количеству:
   $$\text{MachDurationCal} = \frac{\text{MachTotalHrs}}{N_{\text{mach}}}$$
   Этот параметр используется для построения синей пунктирной стрелки на диаграмме Ганта.

## 2. Пошаговый план изменений

### Шаг 1. Расширение модели данных (Интерфейсы)
В файле [fortification-planner.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/fortification-planner/fortification-planner.component.ts) обновить интерфейс `VopTask`:
* Добавить необязательное свойство `machQty?: number;` (количество машин, по умолчанию `1`).

### Шаг 2. Обновление логики автосинхронизации и импорта
* В методе `loadFromMap()` инициализировать `machQty: 1` для всех генерируемых задач.
* В реактивном эффекте автосинхронизации в конструкторе `FortificationPlannerComponent` при создании новых задач указывать `machQty: 1`. При обновлении существующих задач — сохранять уже введенное пользователем количество машин (`machQty: t.machQty || 1`).

### Шаг 3. Изменение алгоритма расчета сроков
В computed-сигнале `vopCalculations()` обновить формулы:
* Получать количество машин: `const mQty = Math.max(1, task.machQty || 1);`.
* Вычислять суммарные машино-часы (ресурс):
  ```typescript
  const machTotal = dev.id !== 'none'
    ? (task.qty * task.machNorm * perfFactor / dev.efficiency)
    : 0;
  ```
* Вычислять календарную продолжительность работы техники для шкалы Ганта:
  ```typescript
  const machDurationCal = dev.id !== 'none' ? (machTotal / mQty) : 0;
  ```
* Заменить использование `machTotal` на `machDurationCal` при расчете временных сегментов и стартовых точек ручного труда:
  ```typescript
  const machEndWork = machStartWork + machDurationCal;
  const manStartWork = task.unit === 'м' ? (machStartWork + machDurationCal * 0.3) : machEndWork;
  ```

### Шаг 4. Обновление пользовательского интерфейса (HTML/CSS)
В [fortification-planner.component.html](file:///F:/Vanya/topos/src/app/features/map-view/components/fortification-planner/fortification-planner.component.html):
* Добавить в таблицу календарного плана новую колонку **«Кол-во машин»** (или «Кол-во техн.») рядом с выбором типа техники.
* Колонка должна содержать компактное числовое поле ввода `<input type="number" min="1" max="10" [value]="row.machQty || 1" (input)="updateTaskField(row.id, 'machQty', $event)">`.
* Ввод должен блокироваться (`disabled`), если выбран ручной труд (`machType === 'none'`).
* В таблице в колонке `Всего м.-ч` продолжать выводить общее количество машино-часов (`machTotal`), чтобы ведомость отражала реальный износ техники.

## 3. Критерии приемки (Acceptance Criteria)
1. **Масштабируемость сроков**: При увеличении количества экскаваторов с 1 до 2 на задаче «Отрывка траншей» длина синей пунктирной стрелки на диаграмме Ганта сокращается ровно в 2 раза, а общая продолжительность I очереди сдвигается влево.
2. **Сохранение общего моторесурса**: Колонка `Всего м.-ч` не меняет своего значения при изменении количества машин (так как объем земляных работ неизменен).
3. **Стабильность пользовательского ввода**: При изменении количества техники данные автоматически сохраняются в `localStorage`, страница не падает, а кастомные задачи продолжают корректно работать.
4. **Успешная компиляция**: Проект собирается без ошибок через `npm run build`.
