## 🚀 3. Проверка сборки и устранение ошибок компиляции (TS4111)

### Контекст и проблема
При запуске сборки проекта через команду `npm run build` компилятор Angular выдал ошибки `TS4111: Property 'none' comes from an index signature, so it must be accessed with ['none']`. Ошибки локализовались в файле [fortification-planner.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/fortification-planner/fortification-planner.component.ts) на строках 290 и 343:
```typescript
const dev = devices[task.machType] || devices.none || { id: 'none', basePerf: 1.5, currentPerf: 1.5, efficiency: 1.00 };
```
Причиной является включенный флаг компилятора TypeScript `noPropertyAccessFromIndexSignature`, требующий обращаться к свойствам объектов с индексной сигнатурой (в данном случае `devices` типа `Record<string, MachDevice>`) только через квадратные скобки `['none']`.

### Реализованное решение
1. **Корректировка обращений к свойствам:**
   - В [fortification-planner.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/fortification-planner/fortification-planner.component.ts) на строке 290 и 343 `devices.none` заменено на `devices['none']`.
2. **Верификация сборки:**
   - Повторно запущена команда `npm run build`. Сборка успешно завершена (`Application bundle generation complete` за 17.29 сек), бандл сгенерирован в `dist/topos`. Ошибок компиляции нет.

### 5. Исправление календарного планировщика фортификаций
- **Проблема рассинхронизации при редактировании:** Метод `changeQty` принимал индекс строки в массиве. Но из-за сортировки и раздельного вывода очередей в шаблоне индекс отфильтрованного массива `vopCalculations().rows` не совпадал с индексом в исходном сигнале `vopTasks()`. В результате при нажатии на плюс/минус в одной строке менялось количество сооружений в другой строке.
- **Решение:** Метод `changeQty` переведен на идентификацию по уникальному `id` сооружения вместо его индекса. Обновление в сигнале выполняется через точечный поиск по `id` (`item.id === id`).
- **Проблема прыгающей нумерации:** В качестве порядкового номера (№) в таблице выводился исходный `row.id` сооружения. При импорте с карты ГИС сооружения фильтровались по наличию на карте (qty > 0), из-за чего нумерация содержала разрывы (например: 2, 5, 7).
- **Решение:** В computed-сигнал `vopCalculations().rows` добавлено динамическое вычисление сквозного порядкового номера `displayIndex` для каждого элемента. В шаблоне вместо `row.id` теперь выводится `row.displayIndex`.

### 7. Оптимизация частоты ресничек на траншее
- **Требование:** Уменьшить количество зубцов (ресничек) бруствера на тактической линии траншеи, приведя его к плотности 1 ресничка на 10 метров.
- **Решение:** В [trench-geometry.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/trench-geometry.service.ts) внедрен кумулятивный алгоритм расстановки штрихов вдоль ломаной линии (Linear Referencing). Вместо фиксированного добавления как минимум 1 реснички на каждый мелкий сегмент (что из-за сглаживания Catmull-Rom разбивало сегменты до длины 2-3 метра и приводило к скученности ресничек), шаг `step = 0.00009` градусов (ровно 10 метров) откладывается нарастающим итогом по всей ломаной. Это гарантирует строго равномерную плотность ресничек ровно 1 штука на каждые 10 метров длины траншеи во всех режимах отрисовки.
- **Миграция существующих объектов:** В [tactical-map.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/tactical-map.service.ts) в метод `loadFromStorage()` внедрен цикл автоматической миграции. При загрузке сохраненных траншей из LocalStorage их геометрия принудительно перегенерируется "на лету" по новому кумулятивному алгоритму, что мгновенно обновило все ранее нарисованные траншеи на карте без необходимости их пересоздания.
