## 🚀 1. Опциональное DEM-ориентирование (Чекбокс «Авто-рельеф»)

### Контекст и проблема
Ранее алгоритм авто-привязки угла знака к склонам горизонталей срабатывал принудительно для всех фортификационных знаков (fort_*). В некоторых случаях оператору требуется наносить знаки в произвольном направлении (например, по тактическому замыслу, вопреки уклону холма), не отключая при этом другие автоматические расчеты.

### Реализованное решение
1. **Переключатель в тулбаре:**
   - В верхний тулбар (map-view.html) после кнопки сглаживания добавлен чекбокс с лаконичной подписью «Авто-рельеф».
2. **Слой состояния (Signals):**
   - В tactical-map.service.ts заведен сигнал isTerrainOrientationEnabled.
   - В map.viewmodel.ts проброшен геттер сигнала и метод управления toggleTerrainOrientation().
3. **Условное перевычисление углов:**
   - При размещении знака по клику на карту и при сбросе после перетаскивания добавлена проверка состояния флага.
   - Если чекбокс снят, автоматический пересчет игнорируется.

## 🚀 2. Исправление сброса сглаживания плавной линии

### Контекст и проблема
При черчении траншеи или хода сообщения превью-линия сглаживалась на лету, если был нажат чекбокс «Плавная кривая». Однако при завершении рисования готовая линия на карте снова становилась угловатой. Это происводило потому, что метод finishDrawingLine() не передавал значение сглаживания в метод сохранения объекта.

### Реализованное решение
1. **Передача флага при фиксации:**
   - В finishDrawingLine() метод сохранения в ViewModel изменен:
     this.tacticalMapService.placeLinearSymbol(coords, mode, name, this.activeLineFlipSide(), this.isLineSmooth());
2. **Результат:**
   - Построенная траншея сразу фиксируется со свойством isSmooth: true (если оно было включено) и отображается со сглаживанием, а свойства объекта в сайдбаре показывают правильный статус чекбокса.

## 🚀 5. Реализация тактических стрелок (Milestone 2.1)

### Контекст и проблема
Для нанесения маневров, направлений главного и вспомогательного ударов, а также путей отхода оператору ГИС требуются динамические тактические стрелки, которые плавно изгибаются по точкам и имеют ГОСТ/МО СССР геометрию (сужение тела, ласточкин хвост для главного удара, пунктирный контур для отхода).

### Реализованное решение
1. **Слои полигонов в MapLibre GL:**
   - В `initLayers()` в [tactical-map.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/tactical-map.service.ts) добавлены слои `tactical_polygons_fill_layer` (тип `fill` с поддержкой прозрачности) и `tactical_polygons_outline_layer` (тип `line` с поддержкой `line-dasharray`).
2. **Математика построения стрелок (Polygon):**
   - В [trench-geometry.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/trench-geometry.service.ts) реализован метод `generateArrowGeometry` для генерации GeoJSON-полигона по контрольным точкам.
   - Поддерживается сглаживание Catmull-Rom для кривой оси при `isSmooth: true`, пропорциональное сужение тела к голове, автоматическое масштабирование наконечника (головы) при малой длине стрелки.
   - Реализована геометрия **«ласточкиного хвоста»** (вдавленный хвост) для стрелки главного удара (`arrow_attack`) и плоского для вспомогательного удара (`arrow_supporting`) / отхода (`arrow_retreat`).
3. **Параметры отрисовки и пунктир:**
   - В `placeLinearSymbol` добавлена выгрузка свойств `fillOpacity` и `lineDashArray` (для `arrow_retreat` используется полупрозрачная заливка `0.25` и пунктир `[3, 3]`).
4. **Кнопки в тулбаре:**
   - В верхний тулбар ([map-view.html](file:///F:/Vanya/topos/src/app/features/map-view/map-view.html)) добавлены 3 кнопки с лаконичными SVG-иконками для рисования стрелок.
   - В `MapViewModel` ([map.viewmodel.ts](file:///F:/Vanya/topos/src/app/features/map-view/viewmodels/map.viewmodel.ts)) зарегистрированы отображаемые имена и обработка выбора/завершения рисования.
5. **Интерактивное редактирование и выбор стрелок:**
   - В `init(map)` в [tactical-map.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/tactical-map.service.ts) слои `tactical_polygons_fill_layer` и `tactical_polygons_outline_layer` включены в перечень `queryRenderedFeatures` при клике по карте. Это открыло возможность выбора стрелок, отображения узлов управления для перетаскивания и удаления.
   - Настроено изменение курсора на `pointer` при наведении на стрелки и траншеи на карте для улучшения UX.

## 🚀 8. Динамическое масштабирование ширины и пропорций тактических стрелок

### Контекст и проблема
Тактические стрелки изначально рендерились с фиксированными геометрическими параметрами и на некоторых масштабах выглядели слишком тонкими («ниточными»). Требовалось увеличить их базовую ширину и дать пользователю инструмент для визуальной трансформации размеров стрелки.

### Реализованное решение
1. **Увеличение базовых размеров:**
   - В [trench-geometry.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/trench-geometry.service.ts) базовые размеры стрелок (ширина хвоста, ширина тела, габариты головы) увеличены примерно в 1.5 раза для лучшей читаемости по умолчанию.
2. **Параметризация ширины в генераторе геометрии:**
   - Метод `generateLinearGeometry()` и внутренний генератор `generateArrowGeometry()` доработаны для приема параметра `lineWidth`.
   - Внедрен коэффициент масштабирования `scaleWidth = lineWidth / 3.0`. Все геометрические параметры стрелки (включая пропорции сжатия) теперь умножаются на этот коэффициент.
3. **Реактивный пересчет геометрии при редактировании:**
   - Метод `updatePlacedSymbolSize()` в [tactical-map.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/tactical-map.service.ts) изменен. Для линейных объектов он сохраняет значение `lineWidth` напрямую (избавились от ошибочного умножения на 40) и принудительно пересчитывает геометрию полигона стрелки в реальном времени.
   - Исправлена проблема со сбросом ширины стрелки обратно в значение `3` при перетаскивании вершин, развороте бруствера и изменении сглаживания. В методах `updateLinearSymbolCoords()`, `toggleSelectedLinearSymbolSide()` и `updatePlacedLineSmooth()` теперь корректно считывается текущий `lineWidth` и передается в `generateLinearGeometry()`.
4. **UX-улучшение панели свойств и расширение лимитов:**
   - Шаблон [symbol-properties.component.html](file:///F:/Vanya/topos/src/app/features/map-view/components/symbol-properties/symbol-properties.component.html) адаптирован: ползунок толщины для стрелок динамически переименовывается в **«Ширина»** (для остальных линий остается **«Толщина»**).
   - Рядом с лейблом выводится текущее числовое значение (например, `Ширина (4.5)`).
   - Максимальный предел толщины/ширины в слайдере увеличен с **10** до **25** (шаг 0.5), что позволяет делать стрелки значительно крупнее для охвата больших тактических зон.
5. **Фиксация толщины обводки полигона (Контур стрелки):**
   - Для слоя `tactical_polygons_outline_layer` в [tactical-map.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/tactical-map.service.ts) толщина линии (`line-width`) зафиксирована на значении `2.5` пикселя. Это предотвращает сильное утолщение контурных границ стрелки при увеличении её геометрических размеров (ранее обводка становилась до 25 пикселей в ширину, полностью перекрывая карту).

## 🚀 9. Сохранение пространственного положения карты при смене подложки

### Контекст и проблема
При переключении базовых карт (например, со Схемы на Топокарту или Спутник) старый инстанс карты полностью уничтожался, а новый создавался с дефолтными координатами (центр Минска, зум 10). Оператору приходилось заново искать свой рабочий район, что существенно замедляло работу.

### Реализованное решение
1. **Сохранение пространственного состояния:**
   - В [map-canvas.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-canvas/map-canvas.component.ts) в методе `renderMap()` перед вызовом `this.map.remove()` считываются текущие координаты центра (`center`), масштаб (`zoom`), угол поворота (`bearing`) и наклон (`pitch`) старой карты.
2. **Передача параметров:**
   - Полученные параметры передаются в конструктор нового экземпляра `maplibregl.Map`. Смена подложек теперь происходит абсолютно бесшовно для оператора — положение, масштаб и ориентация карты сохраняются с точностью до пикселя.

## 🚀 10. Включение/отключение прицела и быстрый переход по координатам

### Контекст и проблема
Для повышения точности позиционирования ГИС оператору требовался визуальный маркер центра экрана (прицел). Также отсутствовала возможность быстрого перемещения карты на конкретные координаты — приходилось вручную прокручивать карту.

### Реализованное решение
1. **Визуальный прицел по центру холста:**
   - В [MapViewModel](file:///F:/Vanya/topos/src/app/features/map-view/viewmodels/map.viewmodel.ts) добавлен сигнал `isCrosshairVisible` и метод `toggleCrosshair()`.
   - В верхний тулбар [map-view.html](file:///F:/Vanya/topos/src/app/features/map-view/map-view.html) встроен чекбокс **«Прицел»**.
   - По центру карты рендерится легкий тактический SVG-прицел (перекрестие с кругом и центральной точкой) красного цвета, не перехватывающий события мыши (`pointer-events: none`).
2. **HUD-координаты под прицелом:**
   - В `MapViewModel` добавлен реактивный сигнал `centerCoords`, хранящий координаты центра карты.
   - В [map-canvas.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-canvas/map-canvas.component.ts) реализован метод `updateCenterCoords()`, который пересчитывает координаты географического центра и обновляет сигнал при инициализации, а также событиях движения (`move`) и масштабирования (`zoom`) карты.
   - Под SVG-прицелом по центру холста рендерится аккуратный плавающий тактический оверлей `.crosshair-coords` с темным полупрозрачным фоном, отображающий текущие координаты наведения (центра экрана) в реальном времени.
3. **Интерактивный ввод и парсинг координат:**
   - Поле координат в статус-баре заменено со статичного текста на текстовое поле ввода `.coords-input` с красивыми шрифтами, ховером и подсветкой фокуса.
   - Метод `parseCoordinates()` во ViewModel распознает координаты в различных форматах (десятичные дроби, направления С.Ш./В.Д. / N/E, а также перестановку широты и долготы местами).
   - Внедрена блокировка: при фокусе в поле ввода (`onCoordsFocus()`) автообновление координат от движения мыши по карте в [map-canvas.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-canvas/map-canvas.component.ts) приостанавливается.
   - Нажатие `Enter` перемещает карту к введенной точке с помощью эффекта `map.flyTo()`, а при ошибке ввода срабатывает CSS-анимация тряски поля (`coordsShake`) с красной подсветкой.

## 🚀 15. Интерактивная схема походного порядка подразделения

### Контекст и проблема
Для правильного планирования марша командиру необходимо строить схему походного порядка подразделения (колонн главных сил и походного охранения: ГПЗ, ГД, ТПО), указывать дистанции между элементами походного порядка, состав сил и средств и типы техники.

### Реализованное решение
1.  **Изменение цвета маркера симуляции и ориентации иконок:**
    - По запросу пользователя цвет тактического грузовика симуляции изменен с синего на красный (`avto1_c_ef4444`, цвет `#ef4444`).
    - **Поворот БМП и БРДМ:** Векторная иконка БМП ([bmp_svoy1.svg](file:///F:/Vanya/topos/public/symbols/bmp_svoy1.svg)) и иконка БРДМ ([brdm1.svg](file:///F:/Vanya/topos/public/symbols/brdm1.svg)) развернуты на 180 градусов с помощью добавления SVG-атрибута `transform="rotate(180, 256, 256)"` к их корневым группам `<g>`.
    - **Новые иконки техники в конструкторе:** В выпадающий список выбора техники подразделения походного порядка марша добавлены специализированные тактические знаки: КНП передвижной (`knp_peredv1`), ПТС (`pts1`), УР (`ur1`), ИМР (`imr1`), КШМ (`kshm4`).
2.  **Служба походного порядка MarchOrderService:**
    - Создан сервис `MarchOrderService` ([march-order.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/march-order.service.ts)), содержащий реактивный сигнал `elements` со стандартной структурой походного порядка мотострелкового батальона на марше.
    - **Новая параметризация подразделения:** Каждое подразделение теперь явно включает: название, тактическую иконку, текстовый состав, количество техники (`vehicleCount`), среднюю длину техники (`vehicleLength`) в метрах (Вариант 3), дистанцию между техникой внутри подразделения (`vehicleDistance`) с единицами измерения (метры/км), дистанцию до следующего подразделения (`distanceToNext`) с единицами измерения.
    - **Расчет глубины колонны (Вариант 3):** Реализован точный метод `getUnitLengthKm(el)` для вычисления физической длины колонны конкретного подразделения с учетом его собственной длины техники (`el.vehicleLength`) и дистанций между ними: $L = N \times L_{\text{техники}} + (N-1) \times D_{\text{внутри}}$. Метод `calculateTotalLengthKm()` суммирует точные физические длины всех подразделений и дистанции между ними для точного вычисления общей глубины колонны батальона на марше.
    - Реализованы методы добавления (`addElement`), удаления (`removeElement`), обновления (`updateElement`), перемещения вверх/вниз для изменения порядка (`moveElement`).
3.  **Пользовательский интерфейс схемы:**
    - В панель `.map-controls` ([map-view.html](file:///F:/Vanya/topos/src/app/features/map-view/map-view.html)) встроена кнопка «Схема походного порядка» со специальной SVG-иконкой многослойного построения.

    - **Визуальная горизонтальная лента:** Отображает походную колонну справа налево (голова колонны слева, движение направлено влево `<---`) с условными иконками техники (БМП, БТР, Танк, Автомобиль, Мед. автомобиль), подписями подразделений, составом сил и соединительными стрелками дистанций, указывающими влево (к голове).
    - **Интерактивная таблица-редактор с Angular CDK Drag-and-drop:** Позволяет интерактивно менять порядок прохождения как с помощью кнопок (▲/▼), так и путем **перетаскивания (Drag-and-drop)** строк. Для устранения любых браузерных блокировок нативного API и перечеркнутых кругов интегрирован официальный модуль `@angular/cdk/drag-drop` версии `21.1.0` (совпадающей по мажорной версии с `@angular/core`). Вся область строк обернута в `cdkDropList` с обработчиком `(cdkDropListDropped)="onCdkDrop($event)"`, строки настроены как `cdkDrag`, а перетаскивание привязано строго к маркеру захвата `⋮⋮` через директиву `cdkDragHandle` для лучшего UX (клик на поля ввода текста не блокирует перетаскивание).
