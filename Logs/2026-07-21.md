### 2. Исправление ошибки `lstat 'F:'` (UNC-пути в Windows)
Устранена ошибка падения Node.js процесса при запуске на Windows с кодом `EISDIR: illegal operation on a directory, lstat 'F:'`:
- **Анализ причины:** Tauri 2 при вызове `app.path().resource_dir()` и `app.path().download_dir()` возвращает канонизированные абсолютные пути с UNC-префиксом `\\?\` (например, `\\?\F:\Vanya\topos\...`). При попытке Node.js разрешить входной скрипт через `resolveMainPath` модуль `fs.realpathSync` некорректно парсит UNC-префикс и пытается выполнить lstat на корневом накопителе как на директории без слэша (`lstat("F:")`), что возвращает системную ошибку `EISDIR`.
- **Решение:** В [lib.rs](file:///F:/Vanya/topos/src-tauri/src/lib.rs) разработана вспомогательная функция `clean_unc_path`, которая отрезает UNC-префикс `\\?\` (первые 4 символа) на Windows. Все пути (`resource_dir`, `download_dir`, `script_path`, `output_path`, `belarus_pmtiles_path`, `topomap_pmtiles_path`), передаваемые дочернему Node.js-процессу или CWD, предварительно прогоняются через `clean_unc_path`, обеспечивая 100% стабильность запуска.

### 3. Ограничение максимального размера иконки (WebGL Native limit)
Устранена критическая ошибка `TypeError: Max height and width is 1024` при попытке загрузить динамические иконки (например, `blindazh`) в `mbgl-renderer` на бэкенде:
- **Анализ причины:** Библиотека `@maplibre/maplibre-gl-native` в своем C++ слое имеет жесткое аппаратное ограничение: максимальный размер добавляемого изображения (иконки) через `map.addImage` не должен превышать 1024x1024 пикселей. Во время экспорта карты высокого разрешения (например, 600 DPI) размер иконок масштабируется на коэффициент разрешения `effectiveRatio`. Если исходный размер иконки велик, результирующий размер превышал 1024 пикселя, вызывая краш C++ биндинга.
- **Решение:** В [map-export.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.ts) (как в ветке SVG-иконок, так и в ветке атласа) внедрено жесткое ограничение: если масштабируемый размер (`targetWidth`/`targetHeight`) превышает 1024 пикселя, он пропорционально сжимается до лимита в 1024 пикселя. Коэффициент `pixelRatio` для этого изображения рассчитывается динамически как `targetWidth / origW`. Это сохранило физический логический размер иконки на карте без искажений и предотвратило любые падения рендерера.

### 4. Исправление предупреждений styleimagemissing (rknp)
Устранены предупреждения и пропуски тактических знаков (например, `rknp`), которые расставлены на карте, но отсутствуют в текущем DOM в виде SVG-элементов:
- **Анализ причины:** При сборе динамических иконок для экспорта фронтенд опрашивает `imageManager` основной карты. Если иконка не представлена в DOM (например, вкладка ее категории в тулбаре закрыта), то срабатывает ветка `else` для копирования сырых пикселей из спрайта. Однако в коде проверялась структура `img.data.width`, предполагая, что `img.data` — это объект `RGBAImage`. В новых версиях MapLibre `img.data` является плоским `Uint8Array`, а ширина и высота лежат на самом объекте `img` (`img.width`, `img.height`), из-за чего иконки пропускались и не попадали в экспорт. В результате фоновый C++ рендерер выдавал варнинг `styleimagemissing`.
- **Решение:** В [map-export.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.ts) внедрен универсальный и безопасный парсер `StyleImage` структуры, который корректно извлекает размеры и сырые пиксели как из плоского `Uint8Array`, так и из вложенного `RGBAImage`:
  ```typescript
  const width = img.width || (img.data ? img.data.width : 0);
  const height = img.height || (img.data ? img.data.height : 0);
  const rawData = img.data ? (img.data.data || img.data) : null;
  ```
  Это позволило успешно собрать все расставленные на карте знаки и передать их в Node.js рендерер.

### 6. Исправление ошибки TypeError: url.endsWith is not a function
Устранена ошибка падения Node-процесса при обработке запросов ресурсов:
- **Анализ причины:** При осуществлении некоторых системных запросов (например, шрифтов или служебных метаданных) библиотека `mbgl-renderer` передает в `requestMock` объект `options`, у которого свойство `url` отсутствует (вместо него используется `uri`) или является сложным объектом (например, инстансом `URL`), а не примитивом `String`. Попытка вызова `url.endsWith` на таких объектах вызывала `TypeError: url.endsWith is not a function`.
- **Решение:** В [index.js](file:///F:/Vanya/topos/src-tauri/sidecar-renderer/index.js) в функции `requestMock` добавлена надежная проверка и безопасное приведение к строковому типу:
  ```javascript
  let url = typeof options === 'string' ? options : (options.url || options.uri);
  if (url && typeof url !== 'string') {
    url = url.toString();
  }
  if (!url) {
    return realRequest(options, callback);
  }
  ```
  Это обеспечило 100% стабильность работы мока на любых сетевых вызовах.

### 7. Исправление ошибки Invalid URI
Устранена критическая ошибка `Error: Invalid URI "/"` в Node.js при обработке запросов к локальным ресурсам с относительными путями:
- **Анализ причины:** В стиле карты шрифты (`glyphs`) или спрайты (`sprite`) часто указываются с относительными путями (например, `/fonts/{fontstack}/{range}.pbf`). В браузере они автоматически разрешаются относительно хоста. Однако в Node.js окружении у `mbgl-renderer` нет домена по умолчанию, из-за чего он делает реальный HTTP-запрос по относительному пути `/` или `/fonts/...`. Это приводило к сбою библиотеки `request` с системным исключением `Invalid URI`.
- **Решение:** В [index.js](file:///F:/Vanya/topos/src-tauri/sidecar-renderer/index.js) в функции `requestMock` реализован перехват относительных путей (которые не содержат протокола и не являются абсолютными Windows-путями). Код пытается разрешить их локально в директории ресурсов (`resourceDir`) или проекта (`baseDir`) в директориях `public/`, `src/` и `src/assets/`. Если файл существует, он считывается и возвращается напрямую через `fs.readFileSync`. В случае отсутствия файла возвращается пустой буфер с кодом `204` (No Content) во избежание падения всего рендерера.

### 10. Исправление TypeError res.request.uri в getRemoteAsset
Устранена критическая ошибка `TypeError: Cannot read properties of undefined (reading 'uri')` при обработке ответов на запросы шрифтов/спрайтов:
- **Анализ причины:** При загрузке ассетов (`kind: glyph` или `kind: sprite`), если возвращается любой статус-код отличный от `200` (например, `204` для отсутствующего шрифта), библиотека `mbgl-renderer` пытается сформировать сообщение об ошибке, обращаясь к `res.request.uri.href`. Так как наши мок-ответы в `requestMock` являлись искусственными объектами без свойства `request`, это приводило к крашу с `TypeError: Cannot read properties of undefined (reading 'uri')`.
- **Решение:** 
  1. Во все моки ответов в `requestMock` добавлено свойство `request: { uri: { href: url } }`.
  2. Для локальных относительных ресурсов, отсутствующих на диске, статус-код ответа изменен с `204` на `200` с пустым буфером `Buffer.alloc(0)`. Это позволяет MapLibre Native продолжить рендеринг (пропустив отсутствующий шрифт), а не ломать экспорт всей карты.

### 11. Исправление ошибки Sharp raw input descriptor (width/height not integer)
Устранена краш-ошибка `Error: Expected width, height and channels for raw pixel input` при генерации PNG в Sharp:
- **Анализ причины:** В `mbgl-renderer/dist/render.js` функция `toPNG` передавала в Sharp параметры `width: width * ratio` и `height: height * ratio`. При вычислении `logicalW = Math.round(targetW / effectiveRatio)` во фронтенде и последующем умножении на дробный `ratio` (например, `300 / 96 = 3.125`), итоговое значение получалось нецелым числом (например, `1746.875`). Валидатор Sharp `is.integer()` отвергал нецелочисленные размеры с фатальным исключением.
- **Решение:** 
  1. В `node_modules/mbgl-renderer/dist/render.js` выражения `width * ratio` и `height * ratio` обернуты в `Math.round()`.
  2. В [map-export.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.ts) добавлена защита `Math.max(1, ...)` при расчете `logicalW` и `logicalH`.

### 12. Исправление рассинхронизации размера буфера VipsImage (Math.floor для Sharp)
Устранена краш-ошибка `Error: VipsImage: memory area too small -- should be X bytes, you passed Y`:
- **Анализ причины:** C++ библиотека MapLibre GL Native вычисляет размер внутреннего пиксельного буфера, преобразуя произведение размеров с помощью отбрасывания дробной части (эквивалент `std::floor` / приведение к `uint32_t`). Использование `Math.round()` на стороне JavaScript приводило к тому, что при определенных дробных `ratio` Sharp ожидал на 1 строку больше пикселей, чем фактически сгенерировал C++ рендерер MapLibre, выпадая с исключением недосдачи байт в буфере.
- **Решение:** В `node_modules/mbgl-renderer/dist/render.js` функция `toPNG` переведена с `Math.round` на `Math.floor`: `width: Math.floor(width * ratio)`, `height: Math.floor(height * ratio)`. Это обеспечило 100% точность байтового размера и полную синхронизацию буферов между MapLibre GL Native и Sharp.
