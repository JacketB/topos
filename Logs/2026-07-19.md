# 📅 2026-07-19: Исправление ошибок дублирования track keys и выборка по расчетам в ведомости района

> **Основной фокус дня:** Устранение предупреждений Angular (NG0955) о дублировании ключей отслеживания при рендеринге списков сооружений, а также интеграция селектора районов непосредственно в модальное окно «Инженерно-фортификационная ведомость района» для обеспечения гибкой выборки при расчете сводных объемов.

---

## 🚀 1. Устранение предупреждений NG0955 (Уникальные ID в track expressions)

### Контекст и проблема
При размещении на карте нескольких однотипных сооружений (например, двух или более крытых ходов сообщения) Angular генерировал предупреждения `NG0955: The provided track expression resulted in duplicated keys for a given collection...` в консоли браузера. Это происходило потому, что шаблон [map-view.html](file:///F:/Vanya/topos/src/app/features/map-view/map-view.html) отслеживал элементы по их названиям (`track item.name`). При возникновении этой ошибки ломался механизм Change Detection, что приводило к рассинхронизации интерфейса (карточки KPI вверху показывали данные по одному району, а спецификация внизу — по другому).

### Реализованное решение
1. **Расширение структуры отчета:**
   - В [fortification-calculation.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/fortification-calculation.service.ts) расширена сигнатура возвращаемого типа `calculateTotalNorms` и добавлен уникальный `id: number` для каждого элемента в массиве `items`.
   - Наполнение массива `items` теперь использует `f.properties?.id` или `f.id` из ГИС-объектов на карте (с резервным автогенератором в виде случайного числа на случай отсутствия `id`):
     ```typescript
     items.push({
       id: f.properties?.id || f.id || Math.floor(Math.random() * 1000000),
       name: norm.name,
       type: typeName,
       earth: norm.earthVolume,
       labor: norm.laborHrs,
       notes: norm.notes
     });
     ```
2. **Обновление track в шаблонах:**
   - В [map-view.html](file:///F:/Vanya/topos/src/app/features/map-view/map-view.html) цикл спецификации переведен на отслеживание по `id`: `@for (item of report.items; track item.id)`.
   - В циклах `report.elementsList` и `report.machinerySummary` в качестве ключа отслеживания теперь используется комбинация имени/типа с индексом `$index` (`track el.name + '_' + $index` и `track m.type + '_' + $index`), что исключает любые дублирования ключей даже при совпадении типов.

## 🚀 2. Интеграция селектора районов в Инженерную ведомость

### Контекст и проблема
Ранее инженерная ведомость рассчитывала данные по району, выбранному в сайдбаре или планировщике, но сама ведомость не имела визуального индикатора или переключателя расчетного района. Это заставляло пользователя совершать лишние клики для смены контекста расчетов.

### Реализованное решение
1. **Интерфейс селектора:**
   - В [map-view.html](file:///F:/Vanya/topos/src/app/features/map-view/map-view.html) в начало контейнера `.fort-report-body` добавлен интерактивный выпадающий список выбора района расчетов, связанный с сигналом `vm.activeCalculationGroupId`:
     ```html
     <div class="fort-report-selector-wrapper">
       <label class="fort-report-selector-label">Район для расчета ведомости:</label>
       <select 
         class="fort-report-selector-select" 
         [ngModel]="vm.activeCalculationGroupId()" 
         (ngModelChange)="vm.activeCalculationGroupId.set($event)"
       >
         <option value="all">Все объекты на карте ({{ vm.placedSymbols().length }} шт.)</option>
         @for (group of vm.objectGroups(); track group.id) {
           <option [value]="group.id">{{ group.name }} ({{ group.elementIds.length }} шт.)</option>
         }
       </select>
     </div>
     ```
   - В компонент `MapView` в файле [map-view.ts](file:///F:/Vanya/topos/src/app/features/map-view/map-view.ts) импортирован модуль `FormsModule` из `@angular/forms` для корректной работы директивы `ngModel`.
2. **Стилизация:**
   - В [map-view.css](file:///F:/Vanya/topos/src/app/features/map-view/map-view.css) добавлены современные стили для `.fort-report-selector-wrapper`, `.fort-report-selector-label`, и `.fort-report-selector-select` с фокусом и тенью в рамках дизайн-системы `Persian Blue`.
3. **Бюджет сборщика:**
   - В связи с ростом объема стилей в `map-view.css` лимит budgets в [angular.json](file:///F:/Vanya/topos/angular.json) для `anyComponentStyle` был увеличен до `maximumWarning: 32kB` и `maximumError: 64kB` для предотвращения прерывания сборки.

## 🚀 5. Исправление математической модели ширины траншеи
- Исправлена принципиальная ошибка в расчете трапецеидального сечения рва: введенная пользователем ширина теперь трактуется как ширина по дну ($W_{bottom}$), а ширина по верху ($W_{top}$) рассчитывается с учетом коэффициентов заложения откосов ($m = 0.25$ для суглинков без укрепления):
  $W_{top} = W_{bottom} + 2 \cdot m \cdot D$.
- Площадь сечения пересчитывается по корректной формуле: $S = \frac{W_{top} + W_{bottom}}{2} \cdot D$.
- Обновлен план тестирования [test_plan.md](file:///F:/Vanya/topos/test_plan.md) (Тест-кейсы 1-3).
- Внесены правки в [fortification-calculation.service.spec.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/fortification-calculation.service.spec.ts) (ожидаемая площадь сечения рва для траншеи 100м шириной по низу 90см и глубиной 110см увеличена с `0.77` до `1.2925` м², ожидаемый объем — до `129.25` м³).

## 🚀 15. Исправление исчезновения икон тактических знаков при экспорте и интерактивный ресайз видоискателя мышью
- **Контекст:** Пользователь выявил 2 недостатка в инструменте экспорта карты:
  1. На сгенерированном PNG-изображении пропадали иконки тактических знаков (оставались только текстовые подписи).
  2. Размеры области кадрирования задавались только через числовые поля ввода в миллиметрах и не позволяли менять рамку мышью прямо на экране.
- **Причина исчезновения икон:**
  В MapLibre GL JS метод `mainMap.getImage(imgId)` возвращает объект `StyleImage` структуры `{ data: RGBAImage, pixelRatio, sdf }`, где сам `RGBAImage` хранит `{ width, height, data }`. Ранее код в `MapExportComponent` пытался извлечь `imgData.width` и `imgData.height` верхнего уровня (которые были `undefined`), передавая невалидный объект в `hiddenMap.addImage()`. В результате MapLibre пропускала отрисовку тактических знаков и выводила только тексты glyphs.
- **Реализация решений:**
  - **Исправление переноса икон:** В [map-export.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.ts) обновлен алгоритм `copyAllImages()`: объект изображения правильно извлекается через `const rawImg = imgData.data || imgData` и передается в `hiddenMap.addImage(imgId, rawImg, { pixelRatio, sdf })`. Скопированные иконки гарантированно регистрируются в фоновой карте при инициализации, событиях `style.load`/`load` и перед вызовом `fitBounds`.
  - **Интерактивный изменение размера видоискателя мышью:**
    * В [map-export.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.ts) реализован метод `startResize(event, handle)` и глобальные обработчики мышью `mousemove` / `mouseup`.
    * Вычисляется динамический масштаб `mmPerPx` и реактивно пересчитываются сигналы `widthMm` и `heightMm` (в диапазоне от 50 до 1200 мм) при перетаскивании маркеров.
    * В [map-export.component.html](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.html) рамка кадрирования оборудована 8 интерактивными маркер-ручками (`nw`, `n`, `ne`, `e`, `se`, `s`, `sw`, `w`) с соответствующими всплывающими подсказками.
    * В [map-export.component.css](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.css) для маркеров заданы стили со свечением, скруглениями, анимацией ховера и соответствующими курсорами изменения размера (`n-resize`, `se-resize` и т.д.).
  - **Тестирование:** В [map-export.component.spec.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.spec.ts) добавлены unit-тесты для верификации работы мышиного ресайза и передачи данных иконок.

## 🚀 16. Независимый ресайз осей рамки экспорта по граням
- **Проблема:** Пользователь заметил, что при перетаскивании верхней/нижней грани рамки изменялся вертикальный размер или сжималась ширина, а при зажатии левой/правой грани менялись оба измерения одновременно.
- **Причина:** Ранее `viewfinderSize` пытался принудительно вписать соотношение сторон `aspectRatio` в максимальный прямоугольник `maxW` × `maxH`. Изменение любой одной стороны меняло `aspectRatio`, что вызывало перерасчет противоположной оси или фиксировало зажатую грань.
- **Решение:** В [map-export.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-export/map-export.component.ts) внедрен computed-сигнал линейного масштаба `scalePxPerMm()`. Экранные размеры рамки `viewfinderSize` переведены на прямые независимые вычисления:
  `width = Math.round(widthMm() * scale)`, `height = Math.round(heightMm() * scale)`.
  Теперь верхние/нижние грани (`n`/`s`) изменяют **исключительно** высоту (`heightMm`), левые/правые грани (`w`/`e`) — **исключительно** ширину (`widthMm`), а угловые маркеры меняют обе оси пропорционально перемещению мыши.

## 🚀 17. Геодезическая калибровка масштабов под широту Беларуси (53.9° N)
- **Проблема:** Пользователь заметил, что топографическая карта отображается не совсем в том масштабе.
- **Причина:** Ранее пресеты зума в `SCALE_PRESETS` ([map-scale.const.ts](file:///F:/Vanya/topos/src/app/features/map-view/consts/map-scale.const.ts)) вычислялись по стандартным формулам Web Mercator для Экватора ($\phi = 0^\circ$). Поскольку Беларусь находится на широте ~53.9° N, искажение проекции Меркатора составляет $\cos(53.9^\circ) \approx 0.589$. В результате выбор масштаба 1:200 000 ставил зум `11.2`, который на самом деле соответствовал масштабу 1:147 600 (~на 35% более крупный растр, чем настоящая двухкилометровка).
- **Решение:**
  - В [map-scale.const.ts](file:///F:/Vanya/topos/src/app/features/map-view/consts/map-scale.const.ts) пересчитаны все пресеты зума для широты 53.9° N на 96 DPI экране (1:200 000 -> `10.77`, 1:100 000 -> `11.77`, 1:50 000 -> `12.77`). При зуме 10.77 1 см на экране ровно равен 2000 м (2 км).
  - В [map-scale.service.ts](file:///F:/Vanya/topos/src/app/features/map-view/services/map-scale.service.ts) метод `getCurrentScale(zoom, lat)` переведен на учет реальной широты центра карты `lat` и стандартов разрешения 96 DPI.
  - В [map-canvas.component.ts](file:///F:/Vanya/topos/src/app/features/map-view/components/map-canvas/map-canvas.component.ts) обновлена передача широты текущего центра карты `this.map.getCenter().lat`.
